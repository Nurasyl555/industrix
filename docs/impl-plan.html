Development Rules & Conventions

Module Structure (Mandatory)

  • Every module lives in modules/<name>/ and MUST contain exactly 5 files: types.go, repository.go, service.go, handler.go, module.go
  • module.go is the entry point. It exposes NewModule(db, redis, ...contracts) *Module that wires: repository → service → handler
  • types.go contains domain models + request/response DTOs. No business logic here.
  • repository.go handles ALL database operations. No SQL outside this file. Use pgx pool directly.
  • service.go contains ALL business logic. Calls repository for data. Implements contracts if needed.
  • handler.go handles HTTP only: parse request → call service → format response. Zero business logic.

Cross-Module Communication (Strict)

  • ✗ NEVER import another module's package: import "modules/identity" — this creates circular deps and tight coupling
  • ✓ ALWAYS use contracts: receive a contracts.UserProvider interface in your service constructor
  • Add new contracts in contracts/contracts.go when a module needs data from another module
  • Contract DTOs must be lightweight read-only structs (e.g. UserBasic{ID, Name, Email}). Never expose internal models.
  • The implementing module maps its internal model → contract DTO at the service boundary

Error Handling

  • Use pkg/errors typed errors: errors.NotFound("user"), errors.Validation("email"), errors.Unauthorized()
  • Repository returns typed errors. Service wraps with context. Handler maps to HTTP status codes.
  • ✗ NEVER return raw error strings to clients. Always use the error type system.
  • Log errors at the handler level with logger.Error().Err(err).Msg("..."). Don't double-log in service + handler.

Database & Migrations

  • One PostgreSQL DB (trust_db), separate tables per module. Module tables are prefixed logically (e.g. marketplace module owns reviews, reputation_scores).
  • Migrations live in backend/migrations/. Numbered sequentially: 001_users.up.sql, 002_companies.up.sql, etc.
  • Always write both .up.sql and .down.sql. Down migration must be reversible.
  • ✗ NEVER use raw SQL in handlers or services. All queries go through the repository layer.
  • Use pgx pool with prepared statements. Use $1, $2 parameterized queries — never string concatenation.

API Design

  • RESTful routes: POST /resource, GET /resource/:id, PUT /resource/:id, DELETE /resource/:id
  • All routes prefixed with /api/v1/. Module routes grouped: /api/v1/companies, /api/v1/deals
  • Public routes (no auth): registration, login, OTP, health, swagger. Everything else requires JWT.
  • Add Swagger annotations to every handler: // @Summary, // @Tags, // @Param, // @Success, // @Router
  • Pagination: use ?page=1&limit=20 query params. Default limit 20, max 100. Return total_count in response.
  • Standard response envelope: {"data": ..., "error": null} or {"data": null, "error": {"code": "NOT_FOUND", "message": "..."}}

Kafka Events

  • Produce events from the service layer (not handler, not repository)
  • Event naming: resource.action — e.g. company.verified, deal.completed, payment.failed
  • Event payload: JSON with event_type, timestamp, actor_id, and resource-specific fields
  • Consumers should be idempotent — handle duplicate deliveries gracefully (use dedup key or "at least once" pattern)

Testing & Quality

  • Unit tests for service layer (mock repository + contracts). Test file: service_test.go next to service.
  • Integration tests for repository layer against real PostgreSQL (use Docker test containers).
  • Handler tests: HTTP-level tests using httptest + Fiber's app.Test().
  • Minimum coverage target: 70% for service layer. Repository and handler coverage is a bonus.
  • Run go vet, golangci-lint on all PRs. Zero warnings policy.

Git & Code Review

  • Branch naming: feat/<module>-<feature>, fix/<module>-<issue>, refactor/<scope>
  • Commit messages: feat(identity): add password reset flow, fix(marketplace): handle nil reputation score
  • PRs must include: description, related issue, test coverage, migration files if DB changes
  • One module per PR when possible. Cross-module PRs require extra review for contract changes.

Quick Reference: Adding New Modules
1. Create modules/<name>/ with 5 files: types.go, repository.go, service.go, handler.go, module.go
2. Write migration in migrations/NNN_<name>.up.sql + .down.sql
3. If module needs data from another module → add interface to contracts/contracts.go
4. Wire in cmd/server/main.go: create module, mount routes
5. Add Swagger annotations to every handler
6. If module emits events → produce to Kafka from service layer
7. NEVER import another module's package directly
