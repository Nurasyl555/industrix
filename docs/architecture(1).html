02 · Platform Layer (Middleware)
No separate API Gateway. All gateway responsibilities (auth, rate limiting, logging, routing) are handled by in-process Fiber middleware. This eliminates gRPC token verification roundtrip and removes an entire service from the deployment.
edge
NGINX
nginx 1.25+
TLS termination, static files, load balancing, WebSocket proxying.
JWT Auth
platform/middleware/auth.go
Direct in-process JWT validation. Injects userID into Fiber context.
Rate Limiter
platform/middleware/ratelimit.go
Redis sliding window. Per-user and per-IP limits.
Request Logger
platform/middleware/logging.go
Structured JSON (zerolog). Method, path, status, latency, trace-id.
▼ In-process function calls (no network hop) → Domain Modules
03 · Domain Modules  — 12 modules · vertical slices · strict isolation via contracts · KZ data residency
Vertical module pattern: Each module is a self-contained slice: types.go → repository.go → service.go → handler.go → module.go. Modules never import each other — cross-module data flows through contracts/ interfaces.
KZ Data Residency: All services run on infrastructure physically located in Kazakhstan (Jusan Cloud / KazTransCom / Yandex Cloud KZ). No personal data transits to foreign SaaS.

04 · Contracts Layer
Cross-Module Interfaces
contracts/contracts.go
UserProvider ← Identity  │  CompanyProvider ← Integrity  │  CatalogProvider (future)  │  ListingProvider (future)  │  PaymentProvider (future)

Modules never import each other. If a module needs data from another, it receives a *Provider interface — not a direct package import. Enables future extraction to microservices.
Shared DTOs
contracts/types.go
UserBasic{ID, Name, Email}
CompanyBasic{ID, Name, BIN, Status}

Lightweight read-only structs returned by contracts. Each module maps its internal models to these DTOs at the contract boundary.

08 · Key Request Flows
Equipment Search (Full Flow)
01
Client sends GET /search?q=crane®ion=almaty&price_max=5M → NGINX → Backend
02
Rate Limit middleware checks Redis → JWT Auth validates token → injects userID
03
Search handler checks Redis cache for identical query hash (TTL 60s)
04
Cache miss → queries OpenSearch with bool query + range filters + geo filter
05
Results cached in Redis, returned to client. Query saved to user history.
Deal Lifecycle (Buyer → Seller)
01
Buyer sends POST /deals referencing listing. Deal module creates deal (status: inquiry)
02
Seller accepts → status: negotiation → parties chat → status: confirmed
03
Deal module calls Payment contract → escrow hold. Booking contract → slot reserved.
04
deal.completed event → Kafka → Review module unlocks review flow
05
Payment escrow released → Notification module fans out confirmation to all channels
